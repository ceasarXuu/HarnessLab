### Summary

第二轮审查发现 **2 个 blocking findings** 和 **7 个 non-blocking risks**。Blocking findings 集中在 `_mark_run_running` 的无条件覆盖（BUG-01 修复未覆盖的竞态窗口）和 `_orphaned_queue_items` 的重复 reconcile（BUG-02 实现与文档不一致）。多个 non-blocking 风险涉及 `wait_experiment_terminal` 的无限挂起、SSE 窗口不足、以及并行 worker 下的事件并发安全。

### Blocking Findings

#### B1: `_mark_run_running` 无条件覆盖 `cancelled` 状态，BUG-01 修复存在未覆盖的 TOCTOU 窗口

**Broken assumption**: "BUG-01 的两阶段 UPDATE 消除了 cancel/result 竞态"

**Failure scenario**:

```text
T1: dequeue_next() 把 runs.status 设为 'running'（BUG-02 修复）
T2: cancel_run() 被调用，把 runs.status 设为 'cancelled'，queue_items.state 设为 'cancelled'
T3: worker.cancel_run() 调用 task.cancel()
T4: _run_one 的 builder.build() 完成（同步操作，CancelledError 尚未抛出）
T5: _mark_run_running() 无条件执行 UPDATE runs SET status='running' WHERE id=?
    → 覆盖了 T2 写入的 'cancelled'，runs.status 变回 'running'
T6: await engine.run(config) 抛出 CancelledError
T7: except CancelledError 块检查 _is_run_cancelled(run["id"])
    → 返回 False（因为 T5 已经把 status 覆盖为 'running'）
T8: _mark_run_interrupted() 把 run 标记为 'interrupted'（而不是 'cancelled'）
```

**Trigger condition**: 用户在 run 被 dequeue 后、`_mark_run_running` 执行前调用 `cancel_run`。

**Likely impact**:
- 用户取消的 run 最终状态为 `interrupted` 而非 `cancelled`，语义错误
- `queue_items.state` 被设为 `cancelled`（T2），但 `runs.status` 最终为 `interrupted`（T8），状态不一致
- 如果 `task.cancel()` 未及时生效（例如 `builder.build` 是 CPU 密集型同步操作），`engine.run` 可能实际开始执行，最终返回 `completed`，覆盖用户取消意图

**Proof needed**:
- `_mark_run_running` 的 UPDATE 语句应加 `AND status NOT IN ('cancelled', 'failed', 'interrupted')` 条件
- 或在 `_mark_run_running` 前检查 `_is_run_cancelled` 并提前返回
- 需要测试覆盖 "dequeue 后、mark_running 前 cancel" 的竞态

**Evidence**:
- `ornnlab/services/experiment_service.py` 第 313-322 行：`_mark_run_running` 的 SQL 无条件 `WHERE id = ?`
- `ornnlab/services/experiment_service.py` 第 258-275 行：`_run_one` 在 `builder.build` 后直接调用 `_mark_run_running`，无状态检查
- `ornnlab/services/experiment_service.py` 第 277-292 行：`CancelledError` 处理依赖 `_is_run_cancelled`，但此时 status 可能已被覆盖

**Plan item affected**: BUG-01 Acceptance Criteria "CancelledError 仍然正确触发取消或 interrupted 路径" — 未完全满足。

#### B2: `_orphaned_queue_items` 缺少 `AND r.status != 'running'` 过滤，导致 running run 被 reconcile 两次

**Broken assumption**: "BUG-02 的 orphan recovery 只处理非 terminal 且非 running 的记录"

**Failure scenario**:

```text
T1: 进程崩溃，runs.status='running', queue_items.state='running'
T2: 进程重启，reconcile_startup() 被调用
T3: _running_runs() 返回该 run（status='running'）
T4: _orphaned_queue_items() 也返回该 run（state='running' 且 status NOT IN terminal）
    → 文档要求加 AND r.status != 'running'，但代码未加
T5: for run in [*running, *orphaned]: _reconcile_run(run) 被调用两次
T6: 第一次 _reconcile_run：读 result.json，写 report，UPDATE runs 为 terminal，append 事件
T7: 第二次 _reconcile_run：result.json 已存在，再次写 report（覆盖），再次 UPDATE runs，再次 append 事件
```

**Trigger condition**: 进程崩溃后重启，存在 `runs.status='running'` 且 `queue_items.state='running'` 的记录。

**Likely impact**:
- 同一个 run 被 reconcile 两次，产生重复的 `experiment.reconcile_decision` 和 `experiment.reconciled` 事件
- report 文件被覆盖写入（虽然内容相同，但 `atomic_write_text` 会重复写）
- `counts` 计数错误：同一个 run 被计数两次（例如 `recovered=2` 而实际只有 1 个 run）
- 如果第一次 reconcile 把 run 标记为 `interrupted`，第二次 reconcile 会再次处理已 terminal 的 run，可能产生不一致状态

**Proof needed**:
- `_orphaned_queue_items` 的 SQL 应加 `AND r.status != 'running'`
- 或在 `reconcile_startup` 中去重（例如按 run_id）
- 需要测试覆盖 "崩溃后重启，running run 不被重复 reconcile"

**Evidence**:
- `ornnlab/services/recovery_service.py` 第 60-68 行：`_orphaned_queue_items` SQL 缺少 `AND r.status != 'running'`
- `ornnlab/services/recovery_service.py` 第 25-36 行：`reconcile_startup` 直接拼接 `[*running, *orphaned]`，无去重
- `docs/releases/v0.1.4/bugfix/02-crash-recovery-blind-spot-dequeue-to-running.md` 第 130-137 行：文档明确要求加 `AND r.status != 'running'`，但实现未遵循

**Plan item affected**: BUG-02 Acceptance Criteria "orphan recovery 的 counts 使用 _reconcile_run 返回值，不把 interrupted 误计为 recovered" — 部分满足，但重复计数问题未覆盖。

### Non-blocking Risks

#### N1: `wait_experiment_terminal` 无超时机制，worker 崩溃时 HTTP 请求无限挂起

**Risk**: `ornnlab/api/experiments.py` 第 28-39 行调用 `worker.wait_experiment_terminal(experiment_id)`，该方法（`worker_service.py` 第 36-46 行）无限循环轮询，没有超时。如果 worker task 崩溃且 run 卡在 `queued`/`running`，HTTP 请求会永远挂起，最终被浏览器/代理超时断开，但服务器端协程泄漏。

**Counterexample**: worker task 因 `last_error` 异常退出，但 run 状态仍为 `running`（如果异常发生在 `engine.run` 之前）。`wait_experiment_terminal` 永远不会看到 terminal 状态。

**Impact**: 资源泄漏，用户体验差。建议添加可选 `timeout_sec` 参数或依赖 worker 健康检查。

---

#### N2: `wait_experiment_terminal` 在 experiment 无 run 时无限等待

**Risk**: `worker_service.py` 第 42-44 行：
```python
statuses = {run["status"] for run in state["runs"]}
if statuses and statuses.issubset(TERMINAL_RUN_STATUSES):
    return
```
如果 experiment 因某种原因没有 run（例如手动删除、数据损坏），`statuses` 为空集，`statuses` 是 falsy，条件不满足，无限循环。

**Counterexample**: 创建 experiment 时 `agent_ids` 或 `benchmark_names` 为空，`run_specs` 为空，experiment 没有 run。

**Impact**: 边界情况下 HTTP 请求挂起。建议 `if not statuses` 时也返回，或检查 experiment 本身状态。

---

#### N3: SSE terminal 检查后的 0.1 秒窗口可能不足以捕获 finalize 事件

**Risk**: `ornnlab/api/experiments.py` 第 155-160 行，SSE 流在检测到 experiment terminal 后 `await asyncio.sleep(0.1)` 再查询剩余事件。如果 `finalize_experiment_if_terminal` 的事件 append 耗时超过 0.1 秒（例如 SQLite busy、磁盘 IO 慢），仍会漏掉最后的 `experiment.{status}` 事件。

**Counterexample**: 并行 worker 下 SQLite busy_timeout=5000，如果多个 run 同时 finalize，`experiment_events` 表写入可能排队超过 0.1 秒。

**Impact**: 前端可能漏掉 experiment 终态事件，但 `stream.end` 事件会携带 status，影响有限。建议改为轮询直到事件 ID 不再增长，或增加重试次数。

---

#### N4: `derive_experiment_status` 优先级顺序可能导致 `failed` 覆盖用户 `cancelled` 意图

**Risk**: `ornnlab/services/experiment_utils.py` 第 33-46 行，优先级为 `completed > partially_failed > failed > cancelled > interrupted > running > queued > draft`。如果 experiment 有 3 个 run：1 个 completed、1 个 failed（自然失败）、1 个 cancelled（用户取消），最终状态为 `partially_failed`，用户取消意图被掩盖。

**Counterexample**: 用户取消 experiment 时，部分 run 已 completed，部分 run 自然 failed，部分 run 被 cancel。`cancel_experiment` 会把非 terminal run 设为 cancelled，但 `finalize_experiment_if_terminal` 会调用 `derive_experiment_status`，`failed` 优先于 `cancelled`，最终状态可能是 `partially_failed` 而非 `cancelled`。

**Impact**: 用户取消语义不明确。建议确认产品语义：cancelled 是否应优先于 failed？

---

#### N5: `_ensured_dirs` 全局变量在测试间累积，非线程安全

**Risk**: `ornnlab/storage/sqlite.py` 第 12 行 `_ensured_dirs: set[str] = set()` 是模块级全局变量。测试中使用多个临时 `settings.home` 时，集合不断增长。多线程下可能重复调用 `ensure_dirs`（虽然幂等）。

**Counterexample**: 测试 A 使用 `/tmp/home-a`，测试 B 使用 `/tmp/home-b`，`_ensured_dirs` 累积两个路径，永远不会清理。

**Impact**: 测试内存泄漏（很小），多线程潜在重复调用。建议改为 `Settings` 实例属性或使用 `functools.lru_cache`。

---

#### N6: 并行 worker 下 `cancel_experiment` 可能错过刚 dequeue 的 run

**Risk**: `ornnlab/api/experiments.py` 第 51-58 行，`cancel_experiment` 先调用 `service.cancel_experiment`（标记所有非 terminal run 为 cancelled），再调用 `worker.cancel_run` 取消 active task。但在并行 worker 下，`cancel_experiment` 遍历 run 列表时，可能有 run 刚被 `dequeue_next` 取出但还未出现在 `_active_runs` 中。

**Counterexample**:
```text
T1: cancel_experiment 读取 run 列表，run X 状态为 'queued'
T2: worker dequeue_next 取出 run X，设为 'running'，创建 task
T3: cancel_experiment 把 run X 设为 'cancelled'（覆盖 'running'）
T4: _mark_run_running 把 run X 设为 'running'（覆盖 'cancelled'）— B1 的竞态
T5: cancel_experiment 调用 worker.cancel_run(run X)，task 存在，被取消
T6: engine.run 抛出 CancelledError，_is_run_cancelled 返回 False（status='running'）
T7: run 被标记为 'interrupted'
```

**Impact**: 与 B1 叠加，取消语义错误。建议 `cancel_experiment` 在标记后重新检查并取消新 dequeue 的 run，或使用队列锁。

---

#### N7: `HarborEngine()` 每个 run 实例化时重复读取环境变量

**Risk**: `ornnlab/services/experiment_service.py` 第 21 行 `self.engine = HarborEngine()`，而 `HarborEngine.__init__` 调用 `_normalize_mode` 读取 `ORNNLAB_HARBOR_ENGINE` 环境变量。在并行 worker 下，每个 run 创建新的 `ExperimentService`，重复读取环境变量。

**Counterexample**: 100 个 run 并行执行，`HarborEngine()` 被实例化 100 次，每次调用 `os.environ.get`。

**Impact**: 性能开销极小，但架构上应缓存。建议 `ExperimentService` 共享 `HarborEngine` 实例，或在 `Settings` 中预解析 mode。

### B2: `_orphaned_queue_items` 缺少 `AND r.status != 'running'` 过滤

**Reason**: 实施修复时复查 `ornnlab/services/recovery_service.py` 第 52-58 行，发现代码已包含 `AND r.status NOT IN ('running', 'completed', 'failed', 'cancelled', 'interrupted')`，即 `'running'` 已在排除列表中。审查时误读了 SQL 字符串。无需修复。

**Validation**: 新增测试 `test_reconcile_startup_does_not_duplicate_running_runs` 验证 running run 不被重复 reconcile，测试通过。

### N1: `wait_experiment_terminal` 无超时机制

### N2: `wait_experiment_terminal` 在 experiment 无 run 时无限等待

**Response**: `accept`

**Action**: 在 `wait_experiment_terminal` 中检查 `if not statuses` 时也返回（experiment 无 run 视为已 terminal）。

### N3: SSE terminal 检查后的 0.1 秒窗口可能不足

**Response**: `defer`

**Reason**: `stream.end` 事件携带 status，前端可据此更新。完全解决需要 event bus 架构，超出 v0.1.4 范围。
