## 审查员输出

### 审查总结
对 5 个文件进行了逐项审查，重点关注代码中显式提出的 5 个问题以及并发安全、事务边界和资源管理。共发现 **2 项阻塞性问题**、**5 项非阻塞风险**，并在测试和可观测性方面存在明显缺口。

---

### 阻塞性发现 (Blocking Findings)

- **[experiment_service.py:371-379]** cursor 在 `with sqlite.connect` 块内创建，但 `cursor.rowcount` 在块外访问。此时 connection 已退出 context manager，若 `sqlite.connect` 在 `__exit__` 中关闭底层连接或清理 cursor 关联资源，`cursor.rowcount` 的可靠性取决于具体实现（标准库 `sqlite3` 通常保留该属性，但并非接口契约）。**更关键的是**：两阶段 UPDATE 的第一阶段（无条件更新 `result_path`/`score`/`harbor_job_id`）与第二阶段（条件更新 `status`）虽然在同一事务，但 `cursor.rowcount == 0` 的检查在事务外——如果 `with` 块执行了隐式 commit，而 cursor 在 connection 关闭后行为未定义，则“完成但被取消”的分支判断可能失效，导致状态机错误地认为 run 已成功更新。应将 `cursor.rowcount` 检查移入 `with` 块内部。

- **[queue_service.py:59-66 + experiment_service.py:308,396-413]** `QueueService.dequeue_next()` 在同一 SQLite 事务中更新了 `queue_items.state='running'` 和 `runs.status='running'`，但紧接着 `ExperimentService._run_one()` 调用 `_mark_run_running()` 独立开启新事务更新 `experiments.status='running'`。若进程/节点在 `dequeue_next` 提交后、`_mark_run_running` 提交前崩溃，`experiments.status` 将永远停留在 `queued`（或更早状态），而 `runs` 和 `queue_items` 已为 `running`。由于 `reconcile_startup` 只扫描 `runs.status='running'`，且 `finalize_experiment_if_terminal` 只在 run 终端后触发，**在 run 真正结束前 experiment 状态长期不一致**，客户端可能误判实验未启动。应将 `experiments.status='running'` 的更新移入 `dequeue_next` 的同一事务，或确保 `_mark_run_running` 与 `dequeue_next` 原子化。

---

### 非阻塞风险 (Non-blocking Risks)

- **[worker_service.py:83,91]** `pending.remove(task)` 在 `done` 集合上迭代执行。当前代码中 `done` 均来源于 `pending` 的子集（通过 `{task for task in pending if task.done()}` 或 `asyncio.wait(pending)`），因此不会 KeyError。**但如果未来有并发逻辑或异常处理逻辑修改了 `pending`（例如通过 `task.add_done_callback` 间接操作）**，`remove()` 会抛出 KeyError。建议将两处 `pending.remove(task)` 改为 `pending.discard(task)`，以消除对集合一致性的隐式假设。

- **[api/experiments.py:137,145]** `event_stream()` 在每次 `while True` 循环迭代中都创建新的 `ExperimentService(settings)` 和 `EventService(settings)` 实例。若 `EventService.__init__` 涉及文件打开、连接建立或缓存初始化，这将造成不必要的资源抖动和潜在的句柄泄漏（取决于 `EventService` 的实现）。应将服务实例的创建移到 `stream()` 函数外部，复用同一实例。

- **[api/experiments.py:161]** `await asyncio.sleep(0.5)` 在客户端断开连接时无法立即响应——必须等待当前 sleep 结束，下一次循环顶部的 `request.is_disconnected()` 才能检测到断开。在终端状态分支中还有额外的 `await asyncio.sleep(0.1)`。这会导致客户端断开后协程仍短暂存活（最多 0.5s+），在高并发 SSE 场景下会累积大量僵尸迭代。建议缩短 sleep 间隔或使用 `asyncio.wait_for(request.is_disconnected(), timeout=0.5)` 的组合来加速退出。

- **[recovery_service.py:56]** `_orphaned_queue_items` 的查询条件存在冗余：`r.status NOT IN ('completed', 'failed', 'cancelled', 'interrupted')` 与 `AND r.status != 'running'` 功能重叠。虽然逻辑正确，但增加了维护成本；若未来新增终端状态（如 `orphaned`），需要同时修改两处，容易遗漏。建议合并为单一 `NOT IN (... 'running')` 子句。

- **[worker_service.py:101-104]** 当 `run is None` 且 `pending` 非空时，`asyncio.wait(..., return_when=asyncio.ALL_COMPLETED)` 返回的 `done` 集合中的任务没有在 `pending` 中显式移除（虽然函数随后 `return`）。这是逻辑不一致——`pending` 集合在该分支结束时仍包含已完成的任务。如果将来在该分支 `return` 前插入使用 `pending` 的代码，会直接引入 bug。

### 缺失的测试

- **两阶段 UPDATE 的 TOCTOU 场景**：模拟引擎完成瞬间，`runs.status` 被并发修改为 `cancelled`，验证 `cursor.rowcount == 0` 分支是否正确触发，且 `result_path`/`score` 不会被错误地标记为有效完成状态。
- **事务边界崩溃恢复**：在 `dequeue_next` 提交后、`_mark_run_running` 之前注入进程崩溃，验证重启后 `experiments.status` 与 `runs.status` 的一致性。
- **SSE 客户端断开**：验证客户端在 `await asyncio.sleep(0.5)` 期间断开后，服务器端协程是否在合理时间内（<1s）退出，不产生事件泄漏。
- **Worker 并发极限**：验证 `limit` 达到时，`asyncio.wait` 返回后新任务能否正确 dequeue，且 `pending` 集合大小始终不超过 `limit`。
- **Recovery 的孤儿队列检测**：构造 `queue_items.state='running'` 但 `runs.status='queued'` 的数据，验证 `_orphaned_queue_items` 能正确检出。

---

### 缺失的日志/可观测性

- **experiment_service._run_one**：`cursor.rowcount == 0` 时虽有 `harbor.job.completed_but_cancelled` 事件，但缺少结构化日志记录实际已写入的 `result_path`/`score` 与最终 `status` 的冲突详情，不利于排查“有结果但状态为取消”的幽灵数据。
- **worker_service 内部状态**：无 `pending` 任务数量、当前并发度、任务完成/失败/取消的 metrics 或日志，无法观察 worker 的实时吞吐和健康度。
- **event_stream 连接生命周期**：缺少 SSE 连接建立和断开的访问日志，无法统计长连接泄漏或客户端异常断开。
- **recovery_service 恢复动作**：`reconcile_startup` 执行了恢复，但缺少按 experiment/run 维度的结构化日志，无法追踪哪些 run 被判定为 `interrupted` 或 `recovered`。
- **状态不一致告警**：`dequeue_next` 与 `_mark_run_running` 分离的事务边界缺乏可观测性埋点，无法提前发现 `experiments` 与 `runs` 状态分叉。

## 主代理响应 (Round 1 Response)

### 阻塞性发现 1: cursor.rowcount 检查位置

**Triaged**: accept (with correction)

**分析**: `sqlite3` 的 `with` 上下文只调用 `commit()`/`rollback()`，不关闭连接，因此 cursor 在块外访问 `rowcount` 在标准库中确实安全。但审查员指出了一个更深层的风险：将 `self.queue.finish()` 和 `self.events.append()` 移入 `with` 块内部会导致**嵌套 SQLite 连接死锁**（`OperationalError: database is locked`）。

**实施修复**:
- 在 `with` 块内仅保存 `cursor.rowcount` 到局部变量 `updated`
- `self.queue.finish()` 和 `self.events.append()` 保持在 `with` 块外执行
- 这样既满足防御性编程要求，又避免嵌套连接冲突

**验证**: `uv run pytest tests/python/` 51 passed, 3 skipped

### 阻塞性发现 2: experiments.status 不在 dequeue 事务中

**Triaged**: accept

**分析**: `dequeue_next` 在同一事务中更新了 `queue_items` 和 `runs`，但 `experiments.status='running'` 在 `_mark_run_running` 中独立事务更新。两者之间存在崩溃窗口，会导致 experiment 状态长期不一致。

**实施修复**:
- `queue_service.py::dequeue_next` 在同一事务中增加 `UPDATE experiments SET status='running'`
- 与 `queue_items` 和 `runs` 的更新共享同一事务，原子化

**验证**: `uv run pytest tests/python/` 51 passed, 3 skipped

### 非阻塞风险 1: pending.remove -> discard

**Triaged**: accept

**实施修复**: `worker_service.py` 中三处 `pending.remove(task)` 改为 `pending.discard(task)`

### 非阻塞风险 2: SSE 服务实例重复创建

**Triaged**: accept

**实施修复**: `api/experiments.py::event_stream` 中将 `ExperimentService` 和 `EventService` 的创建移到 `stream()` 函数外部，循环内复用

### 非阻塞风险 3: SSE sleep 延迟断开检测

**Triaged**: defer

**理由**: 0.5s 延迟在现有使用场景下可接受。高并发 SSE 不是当前版本的主要场景。若未来出现长连接泄漏问题，可引入 `asyncio.Event` 或缩短 sleep 间隔。

### 非阻塞风险 4: recovery 查询条件冗余

**Triaged**: accept

**实施修复**: `recovery_service.py::_orphaned_queue_items` 合并为单一 `NOT IN ('running', 'completed', 'failed', 'cancelled', 'interrupted')`

### 非阻塞风险 5: pending 集合在 run-is-None 分支未清理

**Triaged**: accept

**实施修复**: `worker_service.py` 中 `run is None` 分支的 `for task in done` 循环增加 `pending.discard(task)`

### 缺失测试 / 可观测性

**Triaged**: defer

**理由**: 所有缺失项均为测试覆盖和可观测性增强，不影响当前版本功能正确性。建议纳入后续迭代。
