## Round 1: 初始审查

### 审查输入包

**目标**: 验证 v0.1.4 版本中 9 个 bugfix 的代码实现是否正确、完整、无回归风险。

**审查目标类型**: 代码实现 + 测试覆盖

**目标位置**:
- `ornnlab/services/experiment_utils.py` — BUG-06 状态派生统一
- `ornnlab/services/recovery_service.py` — BUG-02 孤儿恢复 + BUG-06 状态派生
- `ornnlab/services/queue_service.py` — BUG-02 dequeue 原子性
- `ornnlab/services/experiment_service.py` — BUG-01 TOCTOU + BUG-10 dequeue 签名
- `ornnlab/services/worker_service.py` — BUG-10 并行执行 + BUG-09 service 生命周期 + BUG-03 wait 语义
- `ornnlab/settings.py` — BUG-10 worker_max_concurrent 配置
- `ornnlab/storage/sqlite.py` — BUG-08 DB 连接优化
- `ornnlab/services/event_service.py` — BUG-07 追加写 + BUG-04 SSE 辅助方法
- `ornnlab/api/experiments.py` — BUG-04 SSE 实时流 + BUG-03 wait=true API
- `ornnlab/services/harbor_subprocess.py` — BUG-05 subprocess 缺失 result 处理

**变更介绍**:
1. BUG-06: 统一 `derive_experiment_status`，`queued` 优先于 `draft`；删除 recovery_service 中重复的状态派生函数。
2. BUG-02: `dequeue_next` 在同一事务中同时更新 `queue_items.state` 和 `runs.status`；`reconcile_startup` 新增 `_orphaned_queue_items` 扫描。
3. BUG-01: `_run_one` 中 `engine.run` 成功后删除 `_is_run_cancelled` 早退；改为两阶段 UPDATE（保留事实 + 条件更新终态）；新增 `completed_but_cancelled` warning 事件。
4. BUG-10/BUG-09: Worker 改为有界并行执行（`worker_max_concurrent=2`）；`_execute_run` 每 run 独立 `ExperimentService`；`_consume_task_result` 消费 task 异常。
5. BUG-03: `run?wait=true` 改为启动 worker 后调用 `wait_experiment_terminal` 轮询当前 experiment，而非等待全局 idle。
6. BUG-08: `sqlite.connect` 增加 `_ensured_dirs` 缓存；移除重复 `journal_mode=WAL`；新增 `busy_timeout=5000`。
7. BUG-07: `_mirror` 改为 `path.open("a")` 追加写，不再读取重写全量文件。
8. BUG-04: SSE stream 改为持续轮询（0.5s），聚合 experiment + run 事件；terminal 后发送剩余事件和 `stream.end`；支持客户端断开检测。
9. BUG-05: `_read_or_write_result` 在 exit 0 缺失 result.json 时标记 `interrupted` 并附带 `failure_class`/`failure_code`。

**风险重点**:
- 竞态条件：BUG-01 的两阶段 UPDATE 是否真正消除 TOCTOU？`cursor.rowcount` 判断是否有竞态？
- 并发安全：BUG-10 的并行 worker 是否正确处理 SQLite 写竞争？`busy_timeout` 是否足够？
- 状态一致性：BUG-02 的 dequeue 原子更新后，`_mark_run_running` 再次更新是否产生副作用？
- SSE 实时性：BUG-04 的轮询间隔 0.5s 是否会在高并发事件下漏事件或产生乱序？
- 事件系统：BUG-07 的追加写在单进程 asyncio 中是否确实安全？`_mirror` 是否可能在异常时留下损坏行？
- Worker 调度：BUG-10 的 `asyncio.wait` + `pending` set 管理是否有 task 泄漏？
- 回滚/恢复：BUG-02 的 orphan recovery 是否会误恢复本应被用户取消的 run？

**实现完整性关注点**:
- 生产代码路径是否完整？
- 测试是否覆盖所有修改路径？
- 是否有 mock/stub 依赖导致测试通过但生产路径有问题？
